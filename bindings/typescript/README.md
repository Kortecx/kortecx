# `bindings/typescript` — TypeScript / Node.js language binding (napi-rs)

**Status**: structure-only. No source files yet. The TypeScript binding is a
separate piece of work scheduled to land AFTER `kx-executor` (P1.9) makes a
runtime promise meaningful to bind from a host language.

## The rule (restated for TypeScript)

**Bindings depend on core. Core NEVER depends on this binding and stays
unaware that TypeScript / Node.js exists.**

What this means concretely:

- **No `napi`, `napi-derive`, `napi-rs` in any `kx-*` core crate's
  `Cargo.toml`.** Those deps live exclusively in
  `bindings/typescript/Cargo.toml`.
- **No `#[napi]` / `#[napi(constructor)]` / `#[napi(object)]` attributes on
  any type defined in a `kx-*` core crate.** Wrapper types in
  `bindings/typescript/src/` re-expose core types with napi-rs attributes.
- **No "JS / Node is special" carve-outs in core.** If Node needs a different
  serialization shape (`Buffer` vs `Uint8Array` vs `bigint`), that's the
  binding's transform, not core's responsibility.
- **No napi-rs features in the workspace `Cargo.toml`'s
  `[workspace.dependencies]`** for any core crate to use.

## Why napi-rs

When the TypeScript binding lands:

- **[napi-rs](https://napi.rs/)** — the Rust → Node N-API bridge. Compiles
  to a `.node` native addon that loads in CommonJS + ESM. Stable ABI per
  Node's N-API contract; one binary works across Node 18+.
- **Per-platform prebuilt binaries** — napi-rs ships a publish pipeline
  (`@napi-rs/cli`) that produces `.node` files for the matrix
  (linux-x64-gnu, linux-arm64-gnu, darwin-x64, darwin-arm64, win32-x64),
  uploaded as platform-specific npm packages with an optional-deps shim.

## The binding's responsibility surface

The TS wrapper is a **thin transform layer**, not a feature layer. Its job:

1. Re-export core types with idiomatic JS shapes (`MoteId` as a `Buffer`
   or hex `string`; `JournalEntry` as a `{ kind: 'committed', moteId,
   resultRef, … }` object; etc.).
2. Translate Rust `Result<T, E>` to JavaScript errors (napi-rs handles
   this; throws by default).
3. Honor the event loop: long-running core calls (inference, journal fold)
   run on napi's `AsyncTask` background queue so the Node main thread
   stays responsive.
4. Convert between JS values (`Buffer`, `Uint8Array`, `bigint`, plain
   objects) and the bincode canonical shape core uses internally.

Things the wrapper does NOT do:

- It does NOT add features to core. Bindings reflect; they don't extend.
- It does NOT define new domain types. A new domain concept goes in core
  first; the wrapper picks it up after.
- It does NOT cache state. Core owns state; the wrapper holds handles.

## CI

When the binding lands, it gets its own CI job (separate from the core
workspace's `ci.yml`):

- Builds the `.node` artifact via `@napi-rs/cli build --release` on the
  matrix of (linux-x64, linux-arm64, darwin-x64, darwin-arm64, win32-x64) ×
  (Node 18, Node 20, Node 22).
- Runs `node --test bindings/typescript/__test__/` against each artifact.

The job is **independent** of the core CI workflow's `test` job — the
binding's tests CAN run in parallel with core's tests because the
inward-only dependency is preserved.

## Audit invariant — verifiable by command

The inward-only rule is verifiable mechanically. From the workspace root:

```bash
# A: core crates' Cargo.toml MUST NOT mention napi.
grep -rE "napi" kx-*/Cargo.toml && echo "RULE VIOLATED" || echo "ok"

# B: core code MUST NOT contain #[napi(...)] attributes anywhere in src/.
grep -rE "#\[napi" kx-*/src/ && echo "RULE VIOLATED" || echo "ok"

# C: `cargo test --workspace` MUST pass with bindings/ removed.
mv bindings /tmp/bindings.bak && cargo test --workspace && mv /tmp/bindings.bak bindings
```

A future CI job can codify (A) + (B) as an audit step.
